# Evolution Document: PACT (Protocol for Agent Context Transfer)

**Date**: 2026-02-21
**Waves completed**: DISCOVER > DISCUSS > DESIGN > DEVOP > DISTILL > DELIVER

---

## Project Summary

PACT is a git-backed protocol for async human+agent coordination. A local MCP server on each participant's machine wraps git operations into 4 tools, using JSON files in a shared git repo as the transport layer. PACT.md files define request type contracts -- both sender and receiver read the same file.

**The problem**: Agents working in isolated sessions have no structured way to request help from other agents or humans. Context gets lost in copy-paste, Slack threads, and ad-hoc markdown files.

**The solution**: Agent-to-agent email with humans in the loop. An agent packages context into a structured request, pushes it to a shared git repo, and the recipient's agent picks it up, auto-loads the pact, and facilitates the response.

---

## Architecture Overview

**Style**: Modular monolith, ports-and-adapters. 675 lines of production TypeScript across 12 source files.

**4 MCP Tools** (driving ports):

| Tool | Purpose | Git Operations |
|------|---------|----------------|
| `pact_request` | Submit a structured request | write JSON + commit + push |
| `pact_inbox` | Check inbox for pending requests | pull + scan directory |
| `pact_respond` | Respond to a request | write response + git mv to completed + commit + push |
| `pact_status` | Check request status | pull + read files |

**3 Driven Ports** (infrastructure adapters):

| Port | Adapter | Responsibility |
|------|---------|----------------|
| `GitPort` | `git-adapter.ts` (simple-git) | pull, add, commit, push, mv with pull-rebase retry |
| `ConfigPort` | `config-adapter.ts` | Read team config, resolve user identity |
| `FilePort` | `file-adapter.ts` | Read/write JSON, list directories |

**Key properties**:
- Stateless between tool calls -- all state lives in the git repo
- Type-agnostic -- pacts define request types, not server code
- Session-decoupled -- send from session A, receive in session Z months later
- Directory-as-lifecycle: `pending/` -> `completed/` via `git mv`
- Lazy adapter instantiation per tool call

**Tech stack**: TypeScript, simple-git, Zod, @modelcontextprotocol/sdk, Vitest, esbuild

## Key Technical Decisions

### Zod v4 + MCP SDK Compatibility
The @modelcontextprotocol/sdk internally depends on Zod v3-style `.describe()` for tool parameter schemas. Zod v4 changed `.describe()` behavior, causing MCP tool registration to fail silently. The fix was to use `zod/v3` compatibility imports for MCP-facing schemas while using Zod v4 natively for internal validation. This was the single largest time sink (step 04-01).

### esbuild over tsup
tsup was initially planned but esbuild was chosen directly for the single-file bundle. Simpler configuration, faster builds, and no wrapper overhead. The bundle produces `dist/index.js` runnable with plain `node`.

### Lazy Adapter Instantiation
Adapters are created fresh per tool call rather than held as long-lived instances. This avoids stale git state between calls and keeps the server truly stateless. The cost (instantiation overhead) is negligible compared to git operations.

### Unit Test Skipping for Tool Handlers
Steps 02-01 through 03-01 skipped the RED_UNIT phase because acceptance tests provided full behavioral coverage through the driving port with real git repos. Unit tests would have duplicated coverage without adding confidence. Unit tests were written only for foundation code (schemas, ID generation) and MCP wiring.

### Directory-as-Lifecycle (ADR-005)
Request state transitions are directory moves (`pending/` -> `completed/`) rather than status field mutations. This is git-friendly (no merge conflicts on status fields) and makes the lifecycle visible in the filesystem.

## What Went Well

**Outside-In TDD worked as designed.** The 48 acceptance tests written during DISTILL provided clear targets. Enabling them step-by-step during DELIVER kept implementation focused and progress measurable.

**Ports-and-adapters paid off.** The clean separation between tool handlers (driving ports) and infrastructure (driven ports) made each piece independently testable. Acceptance tests exercised the full stack through the driving port without mocks.

**Single-day delivery.** 10 implementation steps from scaffold to distribution in one session. The extensive upfront design (5 waves) meant zero ambiguity during implementation -- every schema, interface, and test scenario was pre-defined.

**Git-based testing.** Using real bare git repos in tests (rather than mocking git) caught real issues like push-rebase timing and atomic commit behavior. The `setup-test-repos.ts` helper made this painless.

**Execution log tracking.** The YAML execution log captured every phase transition with timestamps, enabling this evolution document and providing a clear audit trail.

## What Was Tricky

**Zod v4 / MCP SDK incompatibility.** This was not anticipated during design. The MCP SDK's internal use of Zod v3 `.describe()` patterns broke when Zod v4 changed the API surface. Diagnosing this required reading MCP SDK internals. The `zod/v3` compatibility import was the fix, but it consumed the most time of any single step.

**Step 04-01 duration.** The build configuration step (20:21 - 21:41) was the longest by far, almost entirely due to the Zod compatibility issue above. All other steps completed in under 10 minutes each.

**Step 02-01 required a restart.** The execution log shows a SKIPPED PREPARE for step 02-01 ("already completed in prior attempt"), indicating the step was attempted once and needed re-execution. The second attempt succeeded cleanly.

## Future Considerations

### Tier 2: Brain Service
An optional service watching the git repo for new requests. Could auto-triage, suggest recipients, or pre-load context. The current architecture's stateless design makes this additive -- the brain reads the same repo without requiring changes to the MCP server.

### Tier 3: Institutional Memory
Long-term pattern extraction from completed requests. Which request types are most common, which team members are most responsive for which pacts, what context patterns lead to the best responses. This would build on the git history that PACT already captures naturally.

### Multi-Repo Federation
Currently assumes a single shared repo per team. Federation across repos (cross-team requests) would require a routing layer or convention for cross-repo references.

### Pact Marketplace
The PACT.md pattern is extensible. A shared repository of pact definitions could emerge, letting teams adopt proven request types rather than authoring from scratch.
