# Evolution Document: pact-fmt (Flat-File Pact Store + Group Addressing)

**Date**: 2026-02-24
**Project ID**: pact-fmt
**Scope**: Flat-file pact store, extended catalog metadata, pact inheritance, compressed catalog, scope filtering, group addressing, per-respondent response storage
**Baseline**: Phase 2 Polish at commit 816ddf7

---

## Feature Summary

pact-fmt restructures how PACT defines, discovers, and addresses pacts. The flat-file pact store replaces the `pacts/{name}/PACT.md` directory convention with `pact-store/**/*.md` files using YAML frontmatter for metadata. Single-level inheritance (`extends: parent_name`) lets variant pacts share common definitions. A compressed pipe-delimited catalog format reduces token cost for agent discovery. Group addressing (`recipients: UserRef[]`) enables multi-recipient requests with per-respondent response storage.

**Before**: Pact definitions lived in `pacts/{name}/PACT.md` directories with markdown-table metadata. Discovery returned a flat catalog with name, description, version, and bundle fields. All requests were 1-to-1 (single recipient). Responses stored as `responses/{id}.json` (single file).

**After**: Pact definitions live in `pact-store/**/*.md` with YAML frontmatter carrying scope, defaults, multi_round, attachments, registered_for, and has_hooks. Variants inherit from parents via `extends`. Discovery supports compressed format (`name|description|scope|context_required->response_required`) and scope filtering. Requests can target multiple recipients. Responses stored as `responses/{id}/{user_id}.json` (directory per request). Old formats remain fully readable.

## What Was Delivered

### Phase 01: Flat-File Pact Store and Inheritance (3 steps)

| Step | Name | What It Does |
|------|------|-------------|
| 01-01 | Flat-file pact loader with extended metadata | Recursive glob of `pact-store/**/*.md`, YAML frontmatter parsing, extended fields (scope, defaults, multi_round, attachments, registered_for, has_hooks). Malformed YAML and missing name fields skipped gracefully. |
| 01-02 | Single-level pact inheritance resolution | `extends: parent_name` merges child with parent. Child overrides description/scope/registered_for wholesale. context_bundle.fields shallow-merge. defaults shallow-merge. Deep inheritance rejected, orphan variants excluded. |
| 01-03 | Discover tool integration with flat-file fallback | `pact_discover` uses flat-file loader when `pact-store/` exists, falls back to old `pacts/{name}/PACT.md` format. Extended metadata fields surfaced in catalog entries. |

### Phase 02: Compressed Catalog and Scope Filtering (3 steps)

| Step | Name | What It Does |
|------|------|-------------|
| 02-01 | Compressed pipe-delimited catalog format | `format: "compressed"` returns `name\|description\|scope\|context_required->response_required` entries, one per line. Inherited variants appear as fully resolved entries. |
| 02-02 | Discover tool schema update | `pact_discover` Zod schema updated with optional `format` (enum: full/compressed) and `scope` (string) parameters. Backward compatible. |
| 02-03 | Scope filtering and edge cases | `scope` parameter filters pacts by scope value. No-match returns empty results (not error). Works with both full and compressed formats. |

### Phase 03: Group Addressing with Per-Respondent Responses (3 steps)

| Step | Name | What It Does |
|------|------|-------------|
| 03-01 | Recipients array schema and send handler | `RequestEnvelopeSchema` accepts `recipients: UserRef[]` and optional `group_ref`. Send validates all user_ids exist, sender not in recipients, array non-empty. Old `recipient` field kept for backward compat reading. |
| 03-02 | Per-respondent response storage and inbox filtering | Responses stored as `responses/{id}/{user_id}.json`. Respond verifies caller is in recipients[]. Each respond marks status completed and moves to completed/ (apathy principle -- agents decide whether to wait). Inbox filters by `recipients.some()`. Inbox entries show `recipients_count` and `group_ref`. |
| 03-03 | Status and thread aggregation | `pact_status` reads `responses/{id}/` directory, returns array of per-respondent responses. `pact_thread` aggregates per-respondent responses into thread view. |

### Phase 04: Error Paths and Backward Compatibility (2 steps)

| Step | Name | What It Does |
|------|------|-------------|
| 04-01 | Group validation error paths | Unknown user in recipients fails with identifying error. Empty recipients array fails. Sender in recipients fails. Non-recipient respond rejected. Duplicate response rejected. No files written on error. |
| 04-02 | Backward-compatible readers for old formats | Old `recipient` field coerced to `recipients[]` on read. Response readers check directory first, fall back to single file. Status and thread read both formats. Mixed old/new formats coexist. |

| `src/tools/pact-discover.ts` | Flat-file loader integration, compressed format, scope filtering |
| `src/tools/pact-thread.ts` | Per-respondent response aggregation |

### Structural Pattern

All changes follow the existing ports-and-adapters pattern. No new ports or adapters were introduced. The flat-file loader is a pure function that reads the filesystem through the existing FilePort. Inheritance resolution is a pure transform over loaded pact data. Per-respondent storage uses the existing FilePort and GitPort. The architecture absorbed a significant capability expansion without structural changes.

### Schema Migration

The `RequestEnvelopeSchema` was extended with `recipients[]` (replacing singular `recipient`) and `group_ref`. Read-time coercion handles old envelopes: if `recipient` exists but `recipients` does not, the singular value is wrapped into an array. All new writes use the `recipients[]` format exclusively.

## Key Technical Decisions

### YAML Frontmatter Over Markdown Tables

The original pact format used markdown tables for context_bundle and response_bundle fields. YAML frontmatter is machine-parseable without fragile regex, supports nested structures (defaults, attachments), and allows arbitrary metadata extension. The trade-off is slightly less human-readable at a glance, but agents are the primary readers. See ADR-020.

### Single-Level Inheritance Only

Deep inheritance (grandchild extending child extending parent) is rejected at load time. This keeps the resolution algorithm simple (one pass, no cycle detection needed) and avoids debugging inheritance chains. A child can extend a base; that is sufficient for the target use case of scoped variants (e.g., `request:backend` extending `request`).

### Compressed Catalog as Pipe-Delimited Text

The compressed format reduces token cost for agent discovery by ~70% compared to full JSON catalog entries. The pipe-delimited format (`name|description|scope|context_required->response_required`) was chosen over CSV or JSON-lines because pipes are rare in natural language descriptions and the arrow separator visually distinguishes context from response fields.

### Per-Respondent Directory Storage

Group responses use `responses/{id}/{user_id}.json` rather than appending to a single file. This avoids merge conflicts when multiple recipients respond concurrently (each writes a separate file) and makes the response set browsable in the filesystem. The trade-off is more files, which is negligible at the target scale.

### Each Respond Completes (Apathy Principle)

When any recipient responds, the request moves to `completed/`. PACT does not track how many recipients have responded or wait for all of them. Agents that need all responses can check `pact_status` and see the response directory contents. This follows the apathy principle: PACT stores and delivers, agents coordinate.

### Directory-First Detection Order

For backward compatibility, readers check `responses/{id}/` as a directory first, then fall back to `responses/{id}.json` as a file. Similarly, envelope readers try `recipients[]` first, then coerce singular `recipient` to an array. New writes always use the new format.

## Lessons Learned

### What Went Well

**The apathy audit simplified group semantics.** The original design had PACT tracking group completion state (pending until all respond, then completed). The apathy audit (2026-02-23) removed this enforcement: each respond completes. This eliminated an entire class of state management complexity and the associated edge cases (what if one recipient never responds? what if the team config changes between send and respond?).

**YAML frontmatter was the right call for pact definitions.** Every metadata extension (scope, defaults, multi_round, attachments, registered_for, has_hooks, extends) slotted in naturally. Adding a new field requires no parser changes -- it is just another YAML key. The old markdown table format would have required regex modifications for each new field.

**Backward compatibility was cheaper than expected.** Read-time coercion (singular recipient to array, single file to directory check) added ~20 lines total across 4 files. No migration script needed. Old and new formats coexist transparently.

**Mutation testing drove real quality improvements.** The progression from 74.74% to 80.55% across 3 runs added 54 tests that caught legitimate gaps: assertion specificity on error messages, sort order assumptions, and untested branches in aggregation logic.

### What Was Tricky

**pact-loader.ts accumulated complexity.** This file handles flat-file globbing, YAML parsing, extended metadata extraction, inheritance resolution, compressed catalog generation, and old-format fallback. At 427 mutants tested, it is the densest file in the codebase. A future refactoring could extract inheritance resolution and compressed catalog generation into separate modules.

**Inheritance merge semantics required precise specification.** The first review round flagged that "child overrides parent" was ambiguous for nested structures. The resolution (wholesale replacement for scalars, shallow merge for objects, child-wins on collision) had to be spelled out field by field. This precision prevented implementation ambiguity but required careful AC revision.

**Surviving mutants in defensive code.** 171 mutants survived, primarily in old-format parser branches and fallback paths. These are equivalent mutations (changing a defensive check produces the same observable behavior because the happy path handles it first). Killing them would require testing internal implementation details rather than observable behavior through driving ports.
