# Shared Artifact Registry — GARP MVP

## Purpose

This registry tracks every piece of data that flows between steps across the three
coordination journeys (setup, sender, receiver). Every `${variable}` in TUI mockups
and every field in journey schemas must have a documented source here.

---

## Artifact Index

### A1: GARP Repository

| Property | Value |
|----------|-------|
| **Type** | Git repository (GitHub/GitLab private) |
| **Created by** | Setup journey, step: create-repo |
| **Consumed by** | All journeys, all steps |
| **Lifecycle** | Persistent — exists for the lifetime of the team |
| **Location** | Remote: github.com/{org}/garp-repo. Local: cloned per user |

**Directory structure**:
```
garp-repo/
  config.json                     # A2: Team config
  requests/
    pending/                      # A3: Request files (new)
    active/                       # (reserved for future use)
    completed/                    # A3: Request files (done)
  responses/                      # A4: Response files
  skills/
    {request-type}/
      SKILL.md                    # A5: Skill files
```

---

### A2: Team Configuration

| Property | Value |
|----------|-------|
| **Type** | JSON file |
| **File** | `config.json` |
| **Created by** | Setup journey, step: create-repo |
| **Consumed by** | garp_request (recipient validation), garp_inbox (user identity) |
| **Lifecycle** | Updated when team membership changes |

**Schema** (minimal for MVP):
```json
{
  "team_name": "Acme Support",
  "members": [
    { "user_id": "cory", "display_name": "Cory" },
    { "user_id": "alex", "display_name": "Alex" }
  ]
}
```

**Fields used by**:
- `members[].user_id` -- garp_inbox scans for requests matching this value
- `members[].user_id` -- garp_request validates recipient exists
- `members[].display_name` -- Shown in inbox listings and request displays

### A3: Request File

| Property | Value |
|----------|-------|
| **Type** | JSON file |
| **File** | `requests/{status}/req-{id}.json` |
| **Created by** | Sender journey, step: submit-request |
| **Consumed by** | Receiver journey (inbox, open-request), Sender journey (check-status) |
| **Lifecycle** | pending -> completed (moved by garp_respond) |

**Schema** (rigid envelope, flexible payload):
```json
{
  "request_id": "req-20260221-001",
  "request_type": "sanity-check",
  "sender": {
    "user_id": "cory",
    "display_name": "Cory"
  },
  "recipient": {
    "user_id": "alex",
    "display_name": "Alex"
  },
  "status": "pending",
  "created_at": "2026-02-21T14:00:00Z",
  "deadline": null,
  "context_bundle": { },
  "expected_response": { "type": "text" }
}
```

**Envelope fields** (validated by MCP server):
| Field | Required | Source | Used by |
|-------|----------|--------|---------|
| request_id | yes | Generated by garp_request | All tools (lookup key) |
| request_type | yes | User/agent (matches skill directory name) | garp_inbox (display), receiver (skill auto-load) |
| sender.user_id | yes | GARP_USER env var | garp_inbox (display), response routing |
| sender.display_name | yes | config.json lookup | Display |
| recipient.user_id | yes | User specifies recipient | garp_inbox (filtering), config.json validation |
| recipient.display_name | yes | config.json lookup | Display |
| status | yes | Set by MCP server (pending/completed) | garp_status, directory placement |
| created_at | yes | Generated by garp_request | Display, ordering |
| deadline | no | User-specified | Display (MVP), enforcement (Tier 2) |
| context_bundle | yes | Agent-assembled per SKILL.md | Receiver context loading |
| expected_response | no | SKILL.md guidance | Receiver response composition |

**context_bundle** (flexible, defined by SKILL.md per request type):

For sanity-check type:
| Field | Source | Used by |
|-------|--------|---------|
| customer | Agent extracts from session | Receiver display, investigation context |
| product | Agent extracts from session | Receiver display |
| issue_summary | Agent composes from session | Receiver: primary orientation |
| involved_repos | Agent extracts from session | Receiver: file access |
| involved_files | Agent extracts from session | Receiver: investigation starting points |
| investigation_so_far | Agent composes from session | Receiver: what has already been checked |
| question | User provides (may be prompted by agent) | Receiver: the specific ask |
| zendesk_ticket | Agent extracts or user provides | Receiver: reference |

### A4: Response File

| Property | Value |
|----------|-------|
| **Type** | JSON file |
| **File** | `responses/req-{id}.json` |
| **Created by** | Receiver journey, step: submit-response |
| **Consumed by** | Sender journey, step: check-status / review-response |
| **Lifecycle** | Created once, immutable |

**Schema**:
```json
{
  "request_id": "req-20260221-001",
  "responder": {
    "user_id": "alex",
    "display_name": "Alex"
  },
  "responded_at": "2026-02-21T15:30:00Z",
  "response_bundle": { }
}
```

**Fields**:
| Field | Required | Source | Used by |
|-------|----------|--------|---------|
| request_id | yes | From the original request | Links response to request |
| responder.user_id | yes | GARP_USER env var | Display |
| responder.display_name | yes | config.json lookup | Display |
| responded_at | yes | Generated by garp_respond | Display, audit |
| response_bundle | yes | Agent-composed per SKILL.md | Sender: the answer |

**response_bundle** (flexible, defined by SKILL.md per request type):

For sanity-check type:
| Field | Source | Used by |
|-------|--------|---------|
| answer | Agent composes from investigation | Sender: primary result |
| evidence | Agent composes from investigation | Sender: supporting detail |
| concerns | Agent composes from investigation | Sender: risk awareness |
| recommendation | Agent composes from investigation | Sender: next steps |

### A5: Skill File

| Property | Value |
|----------|-------|
| **Type** | Markdown file |
| **File** | `skills/{request-type}/SKILL.md` |
| **Created by** | Setup journey (initial skills), or any team member adding a new request type |
| **Consumed by** | Sender journey (compose-request), Receiver journey (open-request, compose-response) |
| **Lifecycle** | Versioned in git. Updated by any team member. Synced via git pull. |
| **Distribution** | Automatic via git pull (garp_inbox pulls before scanning) |

**Key property**: ONE skill file per request type, used by BOTH sender and receiver.
This is a correction from the discovery docs which proposed separate sender.md/receiver.md.

**Content structure** (for sanity-check):
```
# Sanity Check

## When to Use
[describes the situation that triggers this request type]

## Composing a Request (Sender)
[fields to gather, how to structure the context bundle]

## Handling a Request (Receiver)
[how to investigate, what kind of response to compose]

## Response Structure
[expected fields: answer, evidence, concerns, recommendation]

## Context Bundle Fields
[field definitions for this request type]
```

**Used as**:
- Sender: Agent reads "Composing a Request" section to know what to gather
- Receiver: Agent reads "Handling a Request" and "Response Structure" to know how to investigate and respond
- Both: "Context Bundle Fields" defines the schema contract

### A6: MCP Source Configuration

| Property | Value |
|----------|-------|
| **Type** | Craft Agents source config (JSON) |
| **File** | Stored in Craft Agents app data (not in GARP repo) |
| **Created by** | Setup journey, step: configure-mcp |
| **Consumed by** | Every tool invocation (MCP server startup) |
| **Lifecycle** | Persistent per Craft Agents installation |

**Key env vars**:
| Variable | Purpose | Source |
|----------|---------|--------|
| GARP_REPO | Absolute path to local repo clone | User sets during setup |
| GARP_USER | User identity for inbox filtering and request signing | User sets during setup |

## Cross-Journey Data Flow

```
SETUP                    SENDER                    RECEIVER
=====                    ======                    ========

A1: Repo created ------> Used by all tools ------> Used by all tools
A2: Config created ----> Recipient validation ---> User identity
A5: Skills created ----> Skill loads for compose -> Skill auto-loads for respond
                    |
                    +--> A3: Request written ----> A3: Request read from inbox
                                                |
                                                +--> A4: Response written
                                                         |
                    A4: Response read <------------------+
```

## Unresolved Design Questions

These surfaced during artifact tracking and need resolution during DESIGN wave:

1. **Request ID generation**: How are IDs generated to be unique across clients?
   Candidates: `req-{YYYYMMDD}-{HHmmss}-{user}`, `req-{uuid}`, `req-{timestamp}-{random}`

2. **Skill auto-load mechanism**: How does the agent know to load `skills/{type}/SKILL.md`?
   The MCP server could include skill content in garp_inbox response, or the agent reads it
   from the local repo path.

3. **Display name resolution**: Should the MCP server resolve user_id to display_name from
   config.json, or should the agent do it?

4. **Context bundle size limits**: What happens if a context bundle is very large? Git handles
   large files poorly. Need a practical size limit.

5. **Request type validation**: Should garp_request verify that `skills/{type}/SKILL.md` exists
   before accepting the request? (Prevents orphaned request types.)
